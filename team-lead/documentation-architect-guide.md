# 文档架构负责人工作指南 / Documentation Architect Guide

> **角色定位**：你负责维护这个项目的 MD 文档体系，使其结构稳定、执行清晰、真源单一。你的职责不是直接产出内容，而是让“写文档的人”和“执行文档的人”都不容易迷路。

---

## 一、你的目标 / Mission

建立一套可持续维护的文档系统，让项目中的每一份 Markdown 都回答清楚三个问题：

1. 这份文档属于什么类型？
2. 谁应该读它？
3. 读完之后应该做什么，或者不该做什么？

---

## 二、你管理的对象 / Scope

你负责的不是某一条内容，而是文档系统本身：

| 类别 | 你要管什么 |
|------|-----------|
| 架构文档 | `framework` 是否稳定、边界是否清楚 |
| 执行文档 | `guide` 是否能直接执行、是否存在歧义 |
| 规划文档 | `plan` 是否与当前目录和真源一致 |
| 报告文档 | `report` 是否格式统一、是否便于复盘 |
| 参考资料 | `reference` 是否被误写成执行文档 |
| 状态文档 | `state` 是否仍然反映项目现状 |

---

## 三、核心原则 / Core Principles

### 3.1 单一真源

- 会变化的项目事实只写在 `team-lead/project-state.md`
- 字段、ID、分类规则只写在对应 `framework`
- 角色怎么执行只写在对应 `guide`

### 3.2 稳定规则与当前状态分离

不要把下面两类信息混写：

- 稳定规则：长期成立，适合写进 `framework` / `guide`
- 当前状态：今天成立，未来可能变化，适合写进 `project-state.md` / `report`

### 3.3 先让人能执行，再追求完整

一份好 guide 的优先级是：

1. 输入清楚
2. 输出清楚
3. 步骤清楚
4. 停止条件清楚
5. 参考资料充分

不是先把背景写满。

### 3.4 文档尽量按角色读取

执行者读 guide，不应该被迫顺手读 5 份 plan 才能开始。

---

## 四、文档类型标准 / Document Type Standard

### 4.1 `framework`

用于定义内容模型。

必须包含：
- 适用范围
- 边界判断
- 分类体系
- ID 规则
- 数据字段

不要包含：
- 当前完成度
- 历史轮次
- 临时目录迁移说明

### 4.2 `guide`

用于驱动一个角色执行。

推荐固定结构：
1. 角色目标
2. 你负责 / 你不负责
3. 开始前先读
4. 输入
5. 输出
6. 工作流程
7. 质量门槛
8. 停止条件
9. 常见错误

### 4.3 `plan`

用于跨角色协调。

必须回答：
- 为什么做
- 分几阶段
- 谁依赖谁
- 最终交付什么

### 4.4 `report`

用于沉淀一次执行结果。

最小结构：
- 标题
- 日期
- 处理范围
- 产出文件
- 结论
- 遗留问题

### 4.5 `reference`

只读资料，不直接下命令。

适合放：
- 搜索关键词库
- 参考表
- 案例库
- 原始 Prompt

### 4.6 `state`

当前真实状态总表。

它不是计划，也不是指南。

---

## 五、审计清单 / Audit Checklist

当你检查一份文档时，按下面顺序：

1. 这份文档的类型是否明确？
2. 标题是否匹配它的真实用途？
3. 是否引用了已经过时的路径或旧目录名？
4. 是否重复硬编码了本该写在 `project-state.md` 的事实？
5. 是否混入了其他文档的职责？
6. 执行者读完后，是否知道输入、输出和停止条件？
7. 是否存在“写了很多，但无法直接执行”的段落？

## 六、常见反模式 / Anti-Patterns

### 6.1 在每份 guide 里重复项目总览

问题：
- 一旦项目规模变化，会到处过期

处理：
- 只保留必要上下文，其余引用 `project-state.md`

### 6.2 把历史状态写成永久规则

问题：
- 例如“当前只有 19 个视频”这类句子很快失效

处理：
- 移入 `state` 或 `report`

### 6.3 把参考资料写进主 guide

问题：
- 让主 guide 过长，执行者难以抓住主线

处理：
- 把大表、关键词库、案例库拆到 `reference/`

### 6.4 输出文件路径不唯一

问题：
- 执行者不知道最终结果该写到哪

处理：
- 在 guide 的“输出”章节用单独表格固定写清楚

### 6.5 用未定义字段或旧字段

问题：
- guide 要求写 `relatedTraining`，framework 却没有这个字段

处理：
- 字段必须先在 framework 定义，再允许 guide 使用

## 七、建议的治理节奏 / Suggested Governance Loop

### 每次新增文档时

1. 先归类文档类型
2. 确认应放目录
3. 检查是否已有同职责文件
4. 再创建

### 每次大改目录或数据结构时

1. 先更新 `project-state.md`
2. 再更新相关 `framework`
3. 再更新相关 `guide`
4. 最后修复 `plan` 和 `report` 中的引用

### 每轮文档清理时

优先级建议：
1. 修旧路径、旧数字
2. 清输入/输出/停止条件
3. 拆大表到 `reference/`
4. 统一 report 模板

## 九、第一阶段改造建议 / Phase 1 Recommendations

当前最优先的不是重写所有文档，而是先完成以下基础治理：

1. 新文档一律先分类，再决定写法。
2. 所有 guide 的开头加上“你负责 / 你不负责 / 输出到哪里”。
3. 所有过时路径统一改成当前真实目录。
4. 总负责人文档改为引用 `project-state.md`，不再硬编码项目规模。
5. 以后凡是想新增“专门说明某个现状”的文档，优先判断能否写进 `project-state.md`。
