# Plan: 攀岩名人堂 / Hall of Fame

> **版本**：v1
> **定位**：这是 Team Lead 产出的高层计划，不是执行细节稿。
> **下一步**：本计划应交给 Task Architect，进一步拆成各 bot 的 task packets。

---

## 1. 需求名称

制作一个可逐步扩展的「攀岩名人堂」主题，并最终接入前端展示。

---

## 2. 本次目标

为网站新增一个独立的「攀岩名人堂 / Hall of Fame」主题，围绕代表性攀岩人物建立可持续扩展的人物档案体系，首轮先完成一小批高代表性人物的内容、视频、图片和前端展示闭环。

本次目标不是一次性完成整个名人堂，而是验证一条稳定可复用的生产链路：

1. 定义人物档案内容结构
2. 产出首批人物内容草案
3. 产出首批人物视频资源草案
4. 产出首批人物图片草案
5. 在前端完成最小可展示版本

---

## 3. 本次范围

### 3.1 主题范围

本次只处理「名人堂」主题，不改动知识库和训练手册的主体结构。

### 3.2 首轮人物范围

建议首轮只做 5 位高代表性人物，覆盖不同历史位置和风格类型：

1. John Gill
2. Lynn Hill
3. Wolfgang Gullich
4. Adam Ondra
5. Janja Garnbret

### 3.3 首轮展示范围

前端只需完成最小可用闭环：

- 一个名人堂列表入口
- 一个名人详情页结构
- 每位人物的基础内容展示
- 每位人物的相关视频展示
- 每位人物的代表图像展示
- 与知识库 / 训练手册的关联入口预留

---

## 4. 本次不做

以下内容不在本轮范围内：

1. 不一次性做完全部名人堂人物
2. 不做复杂的时间线、地图、奖项统计图等增强功能
3. 不做自动化推荐系统
4. 不要求首轮覆盖所有地区或所有代际代表人物
5. 不要求首轮把人物页和训练手册做深度双向联动
6. 不做大规模多语言重写，只要求达到当前站点可用标准

---

## 5. 需要调用的 Bots

### 必需

- [x] Content Bot
- [x] Video Bot
- [x] Image Bot
- [x] Frontend Integrator

### 可选

---

## 6. 各 Bot 的职责边界

### Content Bot

负责：

- 研究首轮 5 位人物档案
- 输出结构化 profile draft
- 标注资料缺口与不确定项

不负责：

- 找视频
- 产出图片
- 修改前端

### Video Bot

负责：

- 为首轮 5 位人物收集高价值视频
- 输出结构化 video draft

不负责：

- 决定前端展示策略
- 代替内容 bot 写人物档案

### Image Bot

负责：

- 为首轮人物准备统一风格的展示图像草案
- 输出图片草案与生成报告

不负责：

- 研究人物事实
- 直接接入前端

### Frontend Integrator

负责：

- 汇总内容、视频、图片 draft
- 形成统一 runtime draft
- 在前端完成最小可用展示结构

不负责：

- 替各研究 bot 补研究结论
- 做最终人工审批

---

## 7. 依赖关系

### 7.1 可并行部分

以下三项可并行：

- 人物内容草案
- 人物视频草案
- 人物图片草案

### 7.2 顺序依赖部分

以下部分必须后置：

1. Frontend Integrator 等待三类 draft 和 self-check 结果汇总
2. Human Final Review 等待 Frontend Integrator 形成整体预览后再进行

### 7.3 总体顺序

```text
Team Lead Plan
  -> Task Architect 拆包
  -> Content / Video / Image 并行执行
  -> Draft Bundle 汇总
  -> Frontend Integrator 整合
  -> Final Review Gate
```

## 9. 首轮建议数据结构

### 9.1 最小内容单位

`profile`

### 9.2 每个 profile 至少包含

1. 基本信息
2. 生涯摘要
3. 代表成就
4. 风格标签
5. 训练理念摘要
6. 采访摘录
7. 视频列表
8. 图片资源
9. 关联知识点
10. 来源

---

## 10. 成功标准

如果以下条件全部满足，则本轮计划视为成功：

1. 首轮 5 位人物全部有结构化 profile
2. 每位人物至少有 1 条高价值视频资源
3. 每位人物至少有 1 张可展示图片
4. 前端已具备名人堂列表页和详情页最小可用版本
5. 操作者可以通过一次 final review gate 对整体结果作出判断

---

## 11. 风险与已知问题

1. 人物资料密度可能极不均匀，尤其是历史人物与当代人物差异很大
2. 视频资源质量可能集中在少数头部人物，导致首轮覆盖不平衡
3. 图片风格如果没有统一标准，容易出现人物之间视觉不一致
4. 如果 Frontend Integrator 拿到的 draft 字段不稳定，会被迫承担本不该承担的整理成本

---

## 12. 对 Task Architect 的交接要求

Task Architect 在拆包时应重点做到：

1. 不增加人工 review 次数
2. 三个执行 bot 的任务范围互不重叠
3. 每个 task packet 都带 self-check 标准
4. 最终收口到一次 final review gate
